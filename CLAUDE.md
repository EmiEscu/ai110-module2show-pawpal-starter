# PawPal+ Project Context

## Role
You are an AI development assistant working on **PawPal+**, a Streamlit web application for a dog care company. Your job is to help design, implement, test, and refine this application. Always keep the UML class diagram, scheduling logic, and Streamlit UI in sync.

## Project Overview
PawPal+ helps pet owners plan and manage daily care tasks for their pets. The app allows users to:
- Register and manage their profile
- Add, update, and remove pets with detailed information
- Track services, subscriptions, and actions tied to each pet
- Define constraints for each pet (time limits, health restrictions, scheduling preferences)
- Maintain a task list of care activities (walks, feeding, meds, grooming, enrichment, etc.)
- Generate a daily plan that schedules tasks based on priorities and constraints
- View clear reasoning for why the plan was arranged a certain way


## UML Class Structure

### Classes and Relationships

1. **User** (Parent / Central Class)
   - Attributes: userId, name, email, phone, pets (list), taskList (list)
   - Methods: addPet(), removePet(), getPets(), addTask(), removeTask(), getTasks(), generateDailyPlan()
   - Owns Pets and Tasks via **composition**
   - Has a **dependency** on DailyPlan (generates it on demand, does not permanently store it)

2. **Pet** (Composed within User)
   - Attributes: petId, name, age, gender, breed, isAble, medications (list), groomingNeeds, constraints (list), actions (list)
   - Methods: addInfo(), updateInfo(), removeInfo(), addMedication(), removeMedication(), addAction(), removeAction(), getActions(), addConstraint(), getConstraints()
   - Owns Actions and Constraints via **composition** (they are deleted if the pet is removed)

3. **Action** (Composed within Pet)
   - Represents subscriptions or services paid for a specific pet (grooming appointments, training, vet visits, etc.)
   - Attributes: actionId, petId, serviceName, serviceType, cost, frequency, status, startDate, endDate
   - Methods: activate(), cancel(), renew(), updateStatus(), isActive()

4. **Constraint** (Composed within Pet)
   - Represents limitations or preferences for a pet (max walk duration, time-of-day restrictions, health-related limits)
   - Attributes: constraintId, petId, constraintType, description, priority, maxDurationMin, timeOfDay
   - Methods: updateConstraint(), getPriority(), isApplicable(task)
   - Has a **dependency** on Task (constraints restrict which tasks can be scheduled and how)

5. **Task** (Composed within User)
   - Represents an individual care activity assigned to a pet
   - Attributes: taskId, title, category, durationMin, priority, status, assignedPetId, dueDate
   - Methods: markComplete(), updatePriority(), updateDuration(), assignToPet(), getDetails()
   - Has a **directed association** to Pet (each task references one pet via assignedPetId)

6. **DailyPlan** (Generated by User on demand)
   - The output of the scheduling logic — a sorted, reasoned schedule for the day
   - Attributes: date, scheduledTasks (list), totalDurationMin, reasoning
   - Methods: addTask(), removeTask(), reorder(), getSummary(), getReasoning()
   - Has an **aggregation** relationship with Task (references existing tasks but does not own them — tasks survive if the plan is deleted)

### Relationship Summary
| From        | To         | Type               | Meaning                                              |
|-------------|------------|--------------------|------------------------------------------------------|
| User        | Pet        | Composition        | User owns pets; pets are deleted with the user        |
| User        | Task       | Composition        | User owns tasks; tasks are deleted with the user      |
| User        | DailyPlan  | Dependency         | User generates a plan on demand, does not store it    |
| Pet         | Action     | Composition        | Pet owns its actions; deleted with the pet            |
| Pet         | Constraint | Composition        | Pet owns its constraints; deleted with the pet        |
| Task        | Pet        | Directed Association | Each task references the pet it is assigned to       |
| DailyPlan   | Task       | Aggregation        | Plan references tasks but does not own them           |
| Constraint  | Task       | Dependency         | Constraints influence which tasks get scheduled       |

## Scheduling Logic
- The scheduler should consider: task priority, task duration, pet constraints (time-of-day, max duration, health), and total available time
- Higher priority tasks should be scheduled first
- Constraints should filter or adjust tasks (e.g., a disabled pet cannot do a long walk)
- The DailyPlan should include a reasoning string explaining why tasks were ordered the way they were
- Edge cases to handle: no tasks, overlapping constraints, total task time exceeding available time
